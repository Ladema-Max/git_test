# git_test
I have developed a non negotiable framework when it comes to the way that I will be handling all my projects going forward.

/*IT IS A 5 PHASE WORKFLOW*/

<!--Phase 1: -->

/*The Problem Brief (5–10 min)*/
What are we solving? Who is it for? What does success look like?
Write 2–3 sentences in plain English. No code words.
Define the "user" (even if it's just you).
List the must-haves vs. nice-to-haves.

<!--Phase 2:-->

/*The Structure Map (10–15 min)*/
What HTML elements do we need? What is the visual hierarchy?
Sketch a rough wireframe on paper or in a text file.
List every section, heading, image, and link.
Decide: semantic HTML or div soup? (Hint: semantic. Always semantic.)

<!--Phase 3:-->

 /*The Style Architecture (10–15 min)*/
How will we organize our CSS? What naming convention?
External CSS file only.
CSS custom properties (variables) for colors, fonts, spacing.
A simple, consistent naming convention.

<!--Phase 4:--> 

/*The Build (Remaining time)*/
Code in small chunks. Test after every chunk.
Build the HTML skeleton first. Open it in the browser. Does the structure make sense without any CSS?
Add CSS in layers: reset → variables → layout → typography → details.
Rule: After every 10–15 minutes of coding, stop and ask: "Would another developer understand this?"

<!--Phase 5:-->
 /*The Review (10 min)*/
Does it match the brief? Is it readable? What would I do differently?
Run through a checklist (I'll provide one each time).
Write a README.md explaining what you built and why you made key decisions.
Reflect: What was harder than expected? What will you reuse?
